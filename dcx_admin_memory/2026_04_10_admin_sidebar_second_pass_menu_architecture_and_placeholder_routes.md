Admin second-pass UX polish moved the shell from a simple first-pass nav into a more believable MVP information architecture. The admin sidebar now uses a recursive nested tree so `Content` can contain deeper sections like `Pages > Categories / Pages` and `Emails > Newsletters / Sequences / Transactional`, while top-level sections now include `Users`, `Schedule`, `Content`, and `Publish`. The older routes remain readable, but the newer content-facing route structure now better matches the product story the client is being shown.

To support that shell shape, `src/App.tsx` now recognizes additional route states and projects them into either existing working screens or believable placeholder surfaces. Working routes now include aliases like `/content/ux`, `/content/emails/newsletters`, and `/content/emails/transactional`, while future-facing routes like `/schedule`, `/content/emails/sequences`, `/content/public`, `/content/app`, and `/content/admin` intentionally land on a shared placeholder page component rather than dead ends. This lets the client click through a coherent admin map without us pretending those deeper mechanics already exist.

In the same pass, the admin content editors started moving onto the actual shadcn control primitives more explicitly. `Button` and `Textarea` are now used in the most visible category/page/newsletter flows, while keeping the DCX-specific editable-border/status-state logic layered on top. The direction is now clear: continue migrating the remaining editable admin surfaces onto the same shadcn `Field` / `Input` / `Select` / `Textarea` / `Button` language, then keep tightening spacing, labels, and corner treatment until the whole admin workspace feels as consistent as the newly polished app surface.
